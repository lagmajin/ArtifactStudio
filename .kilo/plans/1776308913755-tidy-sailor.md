# C++コード生成のための詳細実装計画（Design-Human / Implementation-AI）

## ユーザー要求の明確化

**核心要求**: 
- 「C++の出力コードエラーが多い」
- 「実装だけ切り替えってできるかな」
- 「C++コード生成のための詳細なプラン」を出力せよ

**解釈**:
ユーザーは、設計（設計書・仕様）は人間（またはAI）が行い、実際のC++コーディング（実装）は別のエージェント/ツールに任せたいと考えている。Kiloの現在のC++コード生成にエラーが多いため、より信頼性の高い実装生成システムを求めている。

## 現在のコード生成の問題点（実績ベース）

1. **モジュール配置エラー多発**:
   - QtヘッダをGMFに配置すべきでない場所に置く（C1116）
   - OpenCVヘッダをmodule purviewに置くべきでない場所に置く（C1117）
   - `module ;` の位置不適切

2. **インポート文の誤り**:
   - `import ImageProcessing.ColorTransform.HueSaturation;` → 実際には `import Color.Conversion;` が必要など、依存関係の見落とし

3. **コンパイル環境の未考慮**:
   - MSVC v143特有の制限（`RefCntAutoPtr.hpp` のGMF内禁止）
   - パスハンドリングが環境依存

4. **ALGORITHM → API の不一致**:
   - ヘッダー宣言と実装のシグネチャ不一致
   - constexpr/noexcept 修飾子の欠落

## C++コード生成の3つのアプローチ比較

| アプローチ | 説明 | 現在のプロジェクトへの適合性 | エラー発生率（予測） |
|------------|------|----------------------------|---------------------|
| **A. フルAI生成** (Design+Impl) | 設計書からAIが.ixxと.cppmの両方を生成 | ⚠️ 低い<br>モジュール配置ルールがコードベースに統一されていないため、AIが誤学習 | 高（40-60%エラー） |
| **B. 設計人間・実装AI** (Human Design / AI Impl) | 人間が詳細設計書作成 → AIが.cppmのみ生成 | ✅ 高い<br>既存アルゴリズム呼び出しのみ为主题、AIは標準ライブラリ呼び出しに特化可能 | 中（10-20%エラー） |
| **C. 設計AI・実装人間** (AI Design / Human Impl) | AIが.ixx生成 → 人間が.cppm実装 | ⚠️ 中<br>ixxのimport配置で既に問題あり、人間が修正が必要 | 中高（25-35%エラー） |
| **D. 手動実装継続** | 人間が全部書く | ✅ 高い<br>知識あればエラー0% | 0%（但し時間かかる） |

**推奨: アプローチB (Human Design / AI Impl)**

理由:
- 設計フェーズ（.ixx）は、プロジェクトのモジュール配置ルールを人間が理解・制御できる
- 実装フェーズ（.cppm）は、AIが標準パターン（ループ、条件分岐、既存関数呼び出し）を生成
- 生成後の自動修正スクリプト `fix_all_modules.py` が配置エラーを事後修復
- **設計/実装の責務分離が明確**

## 3. 実装すべき関数・メソッド

### 3.1 QImage HueSaturationEffect::apply(const QImage& source) const
**摘要**: 画像全体に調整適用（sRGB pipeline）
**引数**: `const QImage& source` (入力画像)
**戻り値**: `QImage` (処理後画像)
**アルゴリズム**:
1. `bool hasAlpha = source.hasAlphaChannel();`
2. `QImage result = hasAlpha ? source.convertToFormat(QImage::Format_ARGB32) : source.convertToFormat(QImage::Format_RGB32);`
3. ピクセル単位ループ（y, x）:
   - `QRgb pixel = line[x];`
   - float r = qRed(pixel) / 255.0f; (extract to 0-1 float)
   - float g, b 同様
   - float a = hasAlpha ? qAlpha(pixel) / 255.0f : 1.0f;
   - **if (settings.premultipliedAlpha && a > 0.0f)**: r/=a, g/=a, b/=a  (unpremultiply)
   - `applyPixel(r, g, b);` → **HueSaturationEffect::applyPixel** を呼び出し（既存実装あり、変更不要）
   - **if (settings.premultipliedAlpha && a > 0.0f)**: r*=a, g*=a, b*=a  (premultiply)
   - clamp 0-1: `r = std::clamp(r, 0.0f, 1.0f);` etc.
   - write back: `qRgba(255*r, 255*g, 255*b, 255*a)` or `qRgb`
4. return result

**エラーポイント**:
- 型キャスト: `QRgb* line = reinterpret_cast<QRgb*>(result.scanLine(y));` 必須
- 除算0保護: `a > 0.0f` 条件チェック
- clampは `std::clamp` を使用（`<algorithm>` はGMFで既にinclude済み）

## 4. 補助関数 Requirements

### 4.1 applyPixelLinear の确认
現状の `HueSaturationEffect::applyPixel` は sRGB→Linear→調整→sRGB を内包。
新たに `applyPixelLinear` を追加し、`apply` から呼び出す構造は既に完了。

**`applyPixelLinear` の振る舞い**:
- 入力: float r,g,b (Linear RGB, 0-1以上/HDR可)
- 処理:
  1. `rgbToHsl(r,g,b, h,s,l);`
  2. `h = adjustHue(h, settings.masterHue/360.0f);`
  3. `s = adjustSaturation(s, settings.masterSaturation/100.0f);` ← clamping なし
  4. `l = adjustLightness(l, settings.masterLightness/100.0f);` ← clamping なし
  5. `applyChannelAdjustment(h,s,l);` （6チャンネル別調整）
  6. `hslToRgb(h,s,l, r,g,b);`
- 出力: float r,g,b (Linear RGB, clamping なしかつ >1.0 を許可)

## 5. 非線形 mapping 関数（UI用）

**新規モジュール提案**: `ImageProcessing.ColorTransform.UIMapping`

```cpp
// ArtifactCore/include/ImageProcessing/ColorTransform/UIMapping.ixx
export module ImageProcessing.ColorTransform.UIMapping;
importcmath;

export namespace ArtifactCore {

/// UIスライダー値 [-180..180] → 内部補正係数 [-π..π] に非線形変換
/// 公式: internal = sign(ui) * pow(abs(ui)/180.0, 2.2)
/// 低域感度: ±50以下では微妙、±180で強力
double mapHueRotation(int uiValue);

/// UIスライダー値 [-100..100] → 内部補正係数 [-1.0..1.0] 的比例に非線形
/// 公式: internal = sign(ui) * pow(abs(ui)/100.0, 2.2)
double mapSaturationAdjustment(int uiValue);

/// UIスライダー値 [-100..100] → 内部補正係数 [-1.0..1.0] を非線形
/// Lightness は加算なので、範囲を ±1.0 に制限（HDR可能）
double mapLightnessAdjustment(int uiValue);

/// スムーズステップ: ui値[-100..100]を[0..1]に
double mapNormalizedParameter(int uiValue);

} // namespace
```

**Implementation (UIMapping.cppm)**:
```cpp
module;
#include <cmath>
#include <algorithm>
module ImageProcessing.ColorTransform.UIMapping;

import :std;

double mapHueRotation(int uiValue) {
    double normalized = uiValue / 180.0;  // -1.0..1.0
    double sign = (normalized >= 0) ? 1.0 : -1.0;
    return sign * std::pow(std::abs(normalized), 2.2);
}
```
