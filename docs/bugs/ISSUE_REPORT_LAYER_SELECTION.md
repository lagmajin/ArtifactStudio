# レイヤー選択解除バグについての調査・仮説および修正案

## 概要
Qtのシグナル/スロットから独自の内部イベントシステム(EventBus等)への移行中において、タイムラインウィジェットでレイヤープロパティを編集した際に、意図せずカレントレイヤーの選択が解除され、プロパティウィジェットが`NoLayer`状態になる問題についての調査レポートです。

## 発生経路の仮説
1. **プロパティの編集イベント発行:**
   プロパティウィジェットで値を変更すると、変更を適用するために内部イベント（例: `PropertyChangedEvent` または `LayerUpdatedEvent`）が発行されます。

2. **タイムラインウィジェットの過剰反応:**
   独自のイベントバス（EventBus）経由で `LayerUpdatedEvent` をタイムラインウィジェットが受信します。旧来のQtシグナル/スロット時代は、シグナルブロック（`QSignalBlocker`）等で自身が発生させた変更によるUI再構築を防いでいましたが、独自イベントシステムではこれが適切にブロックまたは判別されていません。

3. **モデルの再構築と選択状態の初期化:**
   タイムラインウィジェットはデータの変更を受け取ると、該当レイヤーを含むツリーやリストのViewModel/UIを完全に再構築・リフレッシュ（Cascade Redraw）してしまいます。この際、現在選択されているアイテムやレイヤーの選択状態(`SelectionManager`側の状態、あるいはUI側の内部フォーカス)を一時退避・復元する処理が漏れているため、選択がクリア（Deselection）されてしまいます。

4. **SelectionChangedイベントの連鎖発行:**
   UIリフレッシュによってタイムラインウィジェット上の選択がクリアされると、今度は `SelectionChangedEvent` (Layer == null) がイベントバスに発行されます。

5. **プロパティウィジェットのNoLayer化:**
   プロパティウィジェットが `SelectionChangedEvent` を受け取り、選択対象が `nullptr` となるため対象レイヤープロパティの表示を破棄し、`NoLayer` 表示へと切り替わります。

## 必要な修正
- **状態保存と復元:** タイムラインウィジェット内でのUI（ツリー/リスト）再構築時に、カレントの選択状態（レイヤーID・UUID等）をキャッシュし、リフレッシュ後に再適用する処理を追加する。
- **イベントの発生源判定と再帰防止:** 独自イベントシステムのペイロードに `SenderId` などを持たせ、自身（プロパティウィジェット）が行った変更起因のイベントであれば、不必要な選択クリアを伴うUIのフルリフレッシュを避ける（または差分更新のみ行う）ようにする。
- **イベントチェーンの抑制:** レイヤーのプロパティ値が変わっただけの場合は `LayerUpdatedEvent` として扱い、レイヤーの追加・削除などの構造変化（`HierarchyChangedEvent`）とは明確に区別し、構造変化時のみ選択状態の再構築ロジックを走らせるようイベントの粒度を調整する。

以上の修正により、プロパティ編集時の不意な選択解除を防ぐことができます。
