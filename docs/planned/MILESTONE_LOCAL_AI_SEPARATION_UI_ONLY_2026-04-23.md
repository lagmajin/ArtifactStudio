# Milestone: Local AI Process Separation (UI Only)

**Status:** Draft
**Goal:** アプリ本体は UI シェルに寄せ、ローカル AI の実行系は別プロセスへ分離する。

---

## 背景

いまの構成では、AI の初期化・推論・メモリ常駐・クラッシュ影響が UI と近すぎる。
これを分離すると、アプリ本体は軽いまま保ちやすくなり、AI 側の失敗を UI に波及させにくくなる。

このマイルストーンでは、アプリを「操作面」として薄く保ち、ローカル AI は別プロセスのサービスとして扱う方針を固める。

---

## 目的

- アプリ本体を UI 専用に近づける
- ローカル AI の実行を別プロセスへ逃がす
- AI 側のクラッシュやハングが UI を巻き込まないようにする
- モデル切り替えや再起動を UI から独立して扱えるようにする
- 将来のリモート AI / ローカル AI の両対応に備える

---

## 非目的

- この段階で AI 機能を全面的に作り直すこと
- モデル選定やプロンプト品質の最適化を先に詰めること
- IPC やプロトコルを過剰に複雑化すること
- UI 側に大きな推論ロジックを残したままにすること

---

## 方針

### 1. UI は薄いシェルにする

- ユーザー入力の受付
- 会話履歴や状態表示
- 実行中 / 待機中 / 失敗の可視化
- ログと診断の入口

### 2. AI は別プロセスに分ける

- モデルロード
- 推論
- ストリーミング応答
- 長時間タスク
- ローカルキャッシュやワーカー管理

### 3. UI と AI の間は純データでつなぐ

- live object を渡さない
- UI の内部状態をそのまま共有しない
- request / response / event の境界を明確にする
- 失敗時は再接続と再試行をしやすくする

---

## 想定構成

### UI プロセス

- 入力フォーム
- 会話一覧
- 状態表示
- エラーメッセージ
- 診断ビュー
- プロセス監視

### AI プロセス

- モデル実行
- トークン生成
- ローカルツール呼び出し
- セッション管理
- ログ出力

### IPC 境界

- JSON ベースの request / response
- ストリーミング event
- セッション ID
- エラーコード
- 再接続ハンドシェイク

---

## フェーズ

### Phase 1: UI Shell 固定

- AI 画面を「操作面」として整理する
- 状態ラベルとログの受け皿を整える
- AI の実体を UI から直接触らない形へ寄せる

### Phase 2: Local AI Worker 分離

- ローカル AI を別プロセス化する
- 起動 / 停止 / 再接続を UI から制御する
- 応答の遅延や失敗を UI に安全に返す

### Phase 3: Protocol 安定化

- request / response schema を固定する
- event 型を絞る
- 互換性のある version field を入れる

### Phase 4: Diagnostics

- AI プロセスの生存確認
- 最終応答時間
- エラー履歴
- 再起動回数
- 通信切断の理由

---

## 完了条件

- アプリ本体が AI の重い処理を直接抱えない
- ローカル AI を別プロセスとして起動できる
- UI だけを再起動しても AI 側を壊しにくい
- AI 側の失敗を UI で説明できる
- 将来のリモート接続へ拡張しやすい境界になっている

---

## リスク

- IPC を早く固めすぎると後から変更しにくい
- プロセス分離で初回応答が少し遅くなる
- UI と AI の責務を曖昧にすると、結局どちらも重くなる
- 診断不足だと「切り離したのに原因が見えない」状態になりやすい

---

## 次の一手

1. UI 側に残す責務を先に確定する
2. AI プロセスの request / response の最小 schema を決める
3. 再接続と失敗表示の方針を固める
